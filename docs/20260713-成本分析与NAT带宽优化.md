# NAT 带宽优化

## 背景

成本分析发现 NAT 网关 7 月 1-13 日已产生 ¥111，是最大开销源，但 TKE VPC-CNI 集群必须保留 NAT 作为唯一出公网通道。通过降带宽优化。

## 网络拓扑

```
VPC 10.0.0.0/16
└── 路由 0.0.0.0/0 → NAT nat-6hvu0buf (EIP 1.15.179.177)
```

NAT 是 TKE Pod 访问上游 API（OpenAI/Anthropic 等）的唯一出口，不可删除。

## 变更

**NAT 网关 EIP 带宽降配：20Mbps → 1Mbps**

- 资源：`eip-1q4hutfv`（绑定 `nat-6hvu0buf`）
- Terraform：`platform/terraform/tencentcloud/nat.tf`
- 变更字段：`tencentcloud_eip.aether_nat.internet_max_bandwidth_out`
- 已验证：`Bandwidth: 1`，`AddressStatus: BIND`

## 效果验证（7/14 复查）

| 指标 | 变更前 (7/1-7/13) | 变更后 (7/1-7/14) | 结论 |
|------|------------------|------------------|------|
| NAT 网关月费 | ¥111.04 | ¥123.04 | 日均 ¥10.25，无变化 |
| EIP 流量费 | ¥0.06 | ¥0.06 | 几乎为零 |

**关键发现：本次降配对成本无影响。**

原因：

1. **NAT 网关实例费为固定单价** `¥0.5/小时`，与带宽无关。无论 1Mbps 还是 20Mbps，只要 NAT 实例存在就按小时计费（约 ¥12/天）。
2. **EIP 按流量计费**（TRAFFIC_POSTPAID_BY_HOUR），整月实际流量仅 `0.000027 GB`，费用可忽略。带宽上限（1Mbps/20Mbps）只限制吞吐速度，不进入计费公式。
3. Terraform 中 NAT 网关 `bandwidth` 参数最小值为 20，无法降低，只能改 EIP 的带宽上限，而 EIP 按流量计费不受带宽上限影响。

## 真正能降本的方向

NAT 网关的成本来自「常驻实例费」，要降本只能：

| 方案 | 预估月费 | 可行性 |
|------|---------|--------|
| 保留 NAT（现状） | ¥360/月满算 | - |
| 改用 TKE 节点绑定 EIP 直出公网 | ¥12/月（EIP流量） | 需改节点网络架构，安全风险 |
| 删除 NAT + 节点直接公网（不推荐） | ¥0 NAT | 暴露节点公网 IP |
| 降 NAT 规格（当前已是小型） | 不可降 | 已是最小规格 |

**结论：当前 NAT 已是最小规格（小型实例），降配 EIP 带宽无法降本。如需进一步降本需评估节点直出公网方案，但会牺牲架构整洁性。**

## 风险与回滚

1Mbps 足够 relay API 请求，但大 payload 或高并发可能成为瓶颈。

回滚：改回 `internet_max_bandwidth_out = 20`，`terraform apply` 即时生效。
