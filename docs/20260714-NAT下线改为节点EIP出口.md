# NAT 网关下线，改为节点 EIP 出公网

## 背景与动机

7 月成本分析发现 NAT 网关是最大开销源（月费约 ¥360），且费用构成是**固定实例费** `¥0.5/小时`（小型实例，与带宽无关），加上几乎可忽略的 EIP 流量费。此前把 NAT 的 EIP 带宽从 20Mbps 降到 1Mbps **对成本无影响**，因为按流量计费的 EIP 只按实际出网流量收费，而 relay 调 LLM API 的 payload 极小（整月流量费约 ¥0.06）。

真正能省钱的唯一方向是**下线 NAT 网关实例**，改用「每个节点绑流量计费 EIP」让 Pod 直连公网。

## 网络拓扑变化

```
变更前:
  VPC 10.0.0.0/16
  └── 路由 0.0.0.0/0 → NAT nat-6hvu0buf (EIP 1.15.179.177, ¥0.5/hr 固定)
        Pod → SNAT → NAT EIP 出公网

变更后:
  VPC 10.0.0.0/16（无默认路由，无 NAT）
  ├── 节点 ins-nd4onzr7 绑 EIP 123.206.123.106（流量计费）
  └── 节点 ins-5x3uwy5j 绑 EIP 124.221.188.134（流量计费）
        Pod → ip-masq-agent SNAT 到节点主网卡 IP → 节点 EIP 出公网
```

## 出网依赖分析

读了所有 k8s manifest 与 terraform：真正必须出公网的只有 **relay 调上游 LLM API（如 OpenRouter）**，流量极小。其余工作负载（account、postgres）只走 cluster 内部。openwebui 当前用公网 URL 调 relay（`OPENAI_API_BASE_URL=https://relay.n0n4w3.cn/v1`，hairpin），不依赖 NAT 出网。

## 机制依据（VPC-CNI / tke-route-eni）

TKE 默认部署 `ip-masq-agent`：把 Pod 出公网流量 SNAT 到节点主网卡 IP。节点主 IP 的出口由「节点是否具备公网能力」决定——绑 EIP 即可走 EIP 出公网，不需要 NAT 网关，也不需要改 VPC 路由表。依据来源：腾讯云官方文档「TKE 容器外网访问能力介绍」与 imroc TKE 实践指南。

## 方案选择

| 方案 | 月省 | 破坏性 | 结论 |
|------|------|--------|------|
| **节点 EIP + ip-masq-agent**（采用） | ~¥360 | 低 | 不重建集群、纯 terraform、保留 VPC-CNI 能力 |
| 切 GlobalRouter + 节点 EIP | ~¥360 | 高（重建集群） | 不必要 |
| Pod 绑 EIP | 负 | 高 | 需集群切固定 IP 模式，当前 `is_non_static_ip_mode=true` 不可用 |
| relay hostNetwork | ~¥360 | 中 | 端口冲突、只解决 relay |
| 保留 NAT（现状） | 0 | 无 | 成本最高 |

## 变更内容（terraform）

`platform/terraform/tencentcloud/`:

- **新增 `node_egress.tf`**：`tencentcloud_eip.node_egress` + `tencentcloud_eip_association.node_egress`，对当前 ASG 返回的两个 worker 节点逐个绑流量计费 EIP（带宽上限 10Mbps，仅作吞吐上限，不影响计费）。
- **重写 `nat.tf`**：删除 `tencentcloud_eip.aether_nat`、`tencentcloud_nat_gateway.aether`、`tencentcloud_route_table_entry.default_internet_via_nat`，保留说明与回滚指引。
- **`outputs.tf`**：移除 `nat_gateway_id` 输出。
- `tke_node_pool.tf` 的 `auto_scaling_config` 维持 `public_ip_assigned = false`：节点 EIP 由 terraform 显式管理，避免 launch-config 自建 EIP 与 terraform EIP 冲突。

## 残余风险与回滚

**未完成的事前验证**：因本地 kubeconfig 是 colima、TKE 公网 API 已关，且 `secrets.env` 为占位符，无法用真实 router-api-key 发起带上游调用的端到端测试，只能确认 relay 进程存活与入站链路正常。**Pod→上游 LLM 的出口链路依赖 ip-masq-agent 机制，需用真实 router-api-key 发一次 chat 请求最终确认。**

**出口 IP 变化**：relay 出口从单一 NAT IP `1.15.179.177` 变为两个节点 EIP（两个 relay 副本各走自己节点）。已与你确认上游 LLM 不依赖出口 IP 鉴权，无影响。

**回滚**：恢复 `nat.tf` 中三个资源定义，`terraform apply` 重建 NAT + 路由条目 + EIP，约 1-2 分钟恢复出公网能力（重建期间 relay 无法调上游）。

**未来节点替换**：node pool 由 ASG 管理，若 ASG 重建某节点，新节点不会自动有 EIP（launch-config 未开 public_ip_assigned）。需在重建后运行 `terraform apply`：`data.tencentcloud_as_instances` 会刷新出新实例 ID，`tencentcloud_eip_association` 会把 EIP 重新绑定到新实例。窗口期该节点上 Pod 无公网出口。

## 后续可选优化
